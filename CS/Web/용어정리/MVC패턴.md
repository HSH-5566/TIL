# MVC패턴
참고링크: https://m.blog.naver.com/jhc9639/220967034588
- 디자인 패턴 중 하나: 유지보수 등을 더 편하게 하기위해 만든 특정한 방법
- MVC: Model, View, Controller의 약자로 하나의 프로젝트, 어플리케이션 구성 시 구성의 요소를 3가지 역할로 구분한 패턴
    - Model
        - 애플리케이션의 정보, 데이터 표현
        - DB, 초기화 값, 변수 등
        - 데이터, 정보들의 가공을 책임지는 컴포넌트
        1. 사용자가 편집하길 원하는 모든 데이터 가지고 있어야함
        2. View, Controller에 대해 어떤 정보도 몰라야함: 데이터 변경 시 Model에서 화면UI 직접 수정 불가
        3. 변경 발생 시 변경 통지 처리방법 구현 필요
    - View
        - UI요소
        - 데이터 및 객체의 입력, 보여주는 출력
        - 데이터를 기반으로 사용자들이 볼 수 있는 화면
        1. Model의 정보를 저장하면 안됨
        2. Model, Controller와 같은 다른 구성요소 몰라야함: View는 데이터 받으면 단순히 출력하는 역할
        3. 변경 발생 시 변경 통지 처리방법 구현 필요
    - Controller
        - 데이터와 UI를 잇는 다리역할
        - 사용자가 데이터 클릭, 수정하는 이벤트를 처리하는 부분
        1. Model, View에 대해 알고 있어야함: Model과 View는 서로 존재를 모르고 변경을 외부로 알리고, 수신하는 방법만을 가지고 있는데 이를 Controller가 중재하기위해 Model과 관련된 View를 알고 있어야 함
        2. Model, View의 변경을 모니터링 해야함: Model, View 변경 통지 받으면 이를 해석해 각 구성요소에게 통지, 애플리케이션의 메인로직은 Controller가 담당
- 사용자가 확인하는 View, Data처리, 이 둘을 중간에서 제어하는 컨트롤로 구성되는 애플리케이션을 만들 때 각각 맡은바에만 집중가능, 효율적, 유지보수성, 확장성, 유연성 증가, 중복코딩 감소
- 단점 : 어플리케이션의 규모가 커질수록 데이터 흐름의 복잡도가 무지막지하게 늘어난다는 것
    - 모델이 늘어날 수록 전파 대상이 늘어나 데이터의 변경사항 신속전파 어려움
    - ex) 하나의 포스트 삭제시 포스트의 댓글도 삭제 필요한데 그 댓글을 사이드 바나 카운터에서 이용하고 있다면 여기에도 변경사항의 전파가 필요
- ex) AngularJS, React(flux패턴도 이용 redux) - View만 신경쓰는 라이브러리, django
- 어떻게 나눌 것인가가 중요한 패턴
